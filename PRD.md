## 2. 产品概述

### 2.1 产品定义

本产品是一个供咨询团队内部使用的本地 AI 能力包。顾问指定本项目需要回应的准则，系统将这些准则合并为统一披露要求，复用同一客户证据库生成覆盖所选准则并集的全面母版粗稿，再按具体准则和客户要求派生适配版本。

MVP 不是独立 SaaS，也不是单一 Prompt。产品由以下部分组成：

- 框架无关的项目数据、规则数据和工作流；
- 文档解析、索引、校验和导出脚本；
- 经专业人员审核的结构化准则库；
- Agent 调用说明和轻量适配层；
- 可跨 Agent 读取的项目目录及中间产物。

### 2.2 核心用户价值

- 客户资料只解析一次，同一证据可回应多个准则条款。
- 相同或相近准则要求只分析一次，保留原条款映射。
- 先生成一份全面母版，再派生不同准则版本，避免维护多份独立稿件。
- 逐条展示回应状态、理由、证据和置信度，方便顾问专业审阅。
- 根据客户的篇幅、颗粒度、模板和参考报告生成不同形式的粗稿。

### 2.3 MVP 成功定义

使用一套脱敏或模拟项目材料，顾问能够完成以下闭环：

> 创建项目 → 确认数据边界 → 确认报告规格 → 指定准则并确认版本 → 解析资料 → 合并准则 → 确认目录 → 生成母版粗稿 → 查看回应矩阵和缺口 → 生成至少一个准则适配稿。

MVP 合格的首要标准是映射准确、证据可追溯、内容不失实。文字润色和运行速度为次要标准。

## 3. 范围

### 3.1 P0 范围

- 自然语言创建项目和确认项目规格；
- 每项目单独确认云端处理、联网、脱敏和文件保留方式；
- Word、PDF、Excel 资料解析及来源定位；
- 顾问指定准则，系统推荐版本，顾问确认后锁定；
- 多准则条款拆解、合并、去重及差异识别；
- 证据与多个统一披露要求、准则条款的复用映射；
- 动态目录和中文母版粗稿；
- 准则回应矩阵、缺口清单、证据清单；
- 准则回应与同业最佳实践双轨评价；
- 特定准则适配粗稿；
- 未确认内容导出拦截；
- 项目状态和中间结果跨 Agent 复用。

### 3.2 P1 范围

- 年份、单位、表文一致性和简单公式校验；
- 报告总篇幅、章节颗粒度和字数预算；
- 已有报告的结构和文风保留；
- 规范披露表格和图表建议；
- 中文稿确认后询问是否生成英文版；
- 中英文术语、指标、年份和口径一致性校核。

### 3.3 不在 MVP 范围

- 正式法律、审计、鉴证或监管结论；
- 完整碳核算、双重重大性分析、气候情景分析；
- 同一客户年度滚动更新；
- 多人实时协作、复杂权限和多级审批；
- 完整视觉设计和无需人工审阅的对外终稿；
- 行业大样本排名；
- OCR、PPT、邮件、网页和压缩包的稳定解析承诺。

## 4. 角色与权限边界

### 4.1 报告顾问

- 创建项目并提供客户材料；
- 指定准则和客户要求；
- 确认数据处理方式、准则版本、报告规格和目录；
- 审阅并确认推断、建议文本和适配内容；
- 导出内部底稿或干净报告。

### 4.2 项目经理或审阅人员

- 审阅回应状态、证据和置信度；
- 修正准则映射、证据映射和正文内容；
- 决定缺口处理优先级；
- 确认试用结果是否达到团队使用要求。

### 4.3 规则库维护人员

- 审核准则来源、条款拆解和统一披露映射；
- 发布不可静默修改的规则版本；
- 保留旧版本和变更记录；
- 不参与具体项目的适用性决定。

MVP 不实现账号或系统权限。上述角色是流程责任，不是登录权限模型。

## 5. 关键术语与枚举

| 术语 | 定义 |
| --- | --- |
| 母版粗稿 | 覆盖本项目所选准则要求并集的内部披露内容母稿 |
| 适配粗稿 | 从母版选择、压缩、重组并补充某一准则特有内容形成的版本 |
| 原始条款 | 准则文件中的原始要求及其编号 |
| 可检查要求 | 从原始条款拆解出的单一、可判断披露要求 |
| 统一披露要求 | 合并多个相同或相近可检查要求后的公共披露单元 |
| 证据 | 客户材料中可定位、可支持报告事实或判断的片段 |
| 项目级要求 | 客户模板、问卷或临时导入要求，仅在当前项目生效 |

### 5.1 内容类型 `content_type`

- `confirmed_fact`：有明确客户证据支持的事实；
- `inference`：由材料间接推导、待人工确认的内容；
- `suggested_text`：基于准则或最佳实践生成的候选文本；
- `information_gap`：缺少事实或证据，不能写入干净终稿。

### 5.2 准则回应状态 `response_status`

- `fully_addressed`：充分回应；
- `partially_addressed`：部分回应；
- `not_addressed`：未回应；
- `not_applicable`：不适用；
- `needs_confirmation`：待确认。

### 5.3 置信度 `confidence`

- `high`：规则、正文和直接证据一致，无明显歧义；
- `medium`：存在间接证据、口径差异或部分缺失；
- `low`：规则解释、证据关联或事实本身存在明显不确定性。

### 5.4 同业评价 `peer_position`

- `leading`：领先；
- `comparable`：同等；
- `lagging`：落后；
- `not_assessed`：未评价。

## 6. 用户主流程

### 6.1 流程状态

项目必须持久化以下状态，Agent 不得只依赖对话记忆：

| 状态 | 含义 | 进入条件 | 可执行下一步 |
| --- | --- | --- | --- |
| `created` | 项目已创建 | 目录初始化完成 | 数据授权确认 |
| `awaiting_data_consent` | 等待数据处理授权 | 项目已创建 | 保存授权结果 |
| `awaiting_spec_confirmation` | 等待项目规格确认 | 已获得数据授权 | 修改或确认规格 |
| `awaiting_standard_confirmation` | 等待准则版本确认 | 规格已确认 | 选择并锁定版本 |
| `ingesting_sources` | 正在解析资料 | 准则版本已锁定 | 查看解析日志 |
| `building_requirement_union` | 正在构建准则并集和账本 | 资料解析完成 | 查看映射与覆盖检查 |
| `awaiting_evidence_confirmation` | 等待证据缺口确认 | 证据库和准则并集已生成 | 确认缺口、映射和冲突证据 |
| `generating_outline` | 正在生成正式目录 | 证据 Checkpoint 已通过 | 查看目录生成状态 |
| `awaiting_outline_confirmation` | 等待正式目录确认 | 正式目录已生成 | 修改或确认目录 |
| `generating_anchor` | 正在生成代表性章节 | 正式目录已确认 | 查看 Anchor 生成状态 |
| `awaiting_anchor_confirmation` | 等待 Anchor 确认 | Anchor 及对应矩阵已生成 | 修改或确认颗粒度、文风和评价方式 |
| `generating_master` | 正在生成完整母版 | Anchor 已确认 | 查看进度和错误 |
| `reviewing_master` | 母版等待审阅 | 母版、矩阵和缺口已生成 | 确认、修改、补件 |
| `adapting_standard` | 正在生成适配稿 | 母版已确认且用户选择目标准则 | 查看适配差异 |
| `awaiting_export_confirmation` | 等待导出前确认 | 所需内部产物已生成 | 处理未确认内容并批准导出 |
| `ready_for_export` | 已通过导出门禁 | 导出 Checkpoint 已通过 | 生成干净版 |
| `blocked` | 任务因缺少关键输入或错误暂停 | 无法继续且不能安全推断 | 补充资料或修改配置 |

#### Checkpoint 持久化对象

`state/workflow.json` 除当前流程状态外，必须保存独立 `checkpoints` 对象：

```json
{
  "workflow_state": "awaiting_anchor_confirmation",
  "checkpoints": {
    "data_consent": {"status": "approved"},
    "project_spec": {"status": "approved"},
    "standards": {"status": "approved"},
    "evidence": {"status": "approved"},
    "outline": {"status": "approved"},
    "anchor": {"status": "awaiting_confirmation"},
    "master": {"status": "pending"},
    "export": {"status": "pending"}
  }
}
```

Checkpoint 状态限定为 `pending`、`ready`、`awaiting_confirmation`、`approved`、`changes_requested`、`blocked`。每个 Checkpoint 还须保存 `updated_at`、可选 `approved_by`、关联产物路径或哈希及备注。只有前置 Checkpoint 为 `approved` 时，状态机才能进入下一阶段。

### 6.2 主流程步骤

1. **创建项目**：生成标准项目目录和唯一 `project_id`，扫描但不外发客户文件。
2. **确认数据边界**：确认云端模型、联网、脱敏、禁外发文件和中间文件保留方式。
3. **确认项目输入与规格**：主动询问客户材料、既有报告或模板、目标准则、优秀案例及报告偏好；优秀案例可以明确不提供，确认结果写入项目状态。
4. **锁定准则版本**：顾问指定准则，系统按报告期间推荐版本，顾问确认后锁定。
5. **构建证据库**：按文件哈希解析资料，保留文件、页码、工作表和单元格位置。
6. **构建准则并集**：加载已锁定规则，形成可检查要求和统一披露要求。
7. **确认证据与缺口**：顾问确认关键缺口、未经审核映射和冲突证据。
8. **生成并确认正式目录**：展示章节目标、篇幅、准则覆盖、证据覆盖和预计缺口。
9. **生成并确认 Anchor**：完成一个代表性章节及对应矩阵，确认颗粒度、文风、内容标记和评价方式。
10. **生成完整母版与矩阵**：按已确认 Anchor 生成其余章节，同时维护内容、要求和证据关联。
11. **人工审阅与补件**：确认或拒绝推断与建议，修正映射，处理缺口。
12. **生成准则适配稿**：按目标准则对母版执行保留、压缩、重组和补充。
13. **导出前确认**：检查未确认内容、账本和业务交付物一致性。
14. **导出**：默认生成一份联合母版 Markdown 和每个已确认准则各一份适配 Markdown；既有 Word/Excel 作为可选内部派生输出，干净版仍受 Checkpoint 约束。

## 7. 详细功能规格

### 7.1 项目启动、授权与规格

#### FR-01 项目创建与规格确认

- 从用户说明和材料中形成项目规格草案；
- 缺少关键字段时只询问会影响结果的事项；
- 保存确认时间和配置版本；
- 规格未确认时不得开始报告生成。

项目规格至少包含主体、报告期间、报告类型、用途、读者、语言、篇幅、颗粒度、目录约束、文风、图表要求、缺口处理方式和交付物。输入确认还必须记录客户材料、既有报告或模板、顾问指定准则，以及优秀案例为“已提供”或“明确不提供”。

#### FR-02 数据处理授权

- 授权记录按项目保存，不跨项目自动继承；
- 禁止云端处理时，任何外部模型调用不得包含客户原文；
- 禁止联网时，不执行同行联网检索；
- 删除中间文件必须由用户明确触发并记录结果。

### 7.2 文件解析与证据

#### FR-03 文档解析

- 支持 `.docx`、文本型 `.pdf`、`.xlsx`；
- Word 提取标题、段落和表格；PDF 保留页码；Excel 保留工作表、单元格区域、显示值和公式；
- 每个文件生成哈希，相同哈希复用解析结果；
- 扫描 PDF 标记为 `unsupported_or_ocr_required`，不得静默输出空文本。

#### FR-06 证据检索与复用

- 证据与要求为多对多关系；
- 证据关系包括 `direct`、`supporting`、`contradicting`；
- 冲突证据必须提示，不得选择性忽略；
- 证据至少定位到文件与页码，或工作表与单元格区域。

### 7.3 准则与映射

#### FR-04 准则选择与版本确认

- 顾问指定准则名称；
- 系统依据报告期末日期推荐版本并展示生效日期、来源和审核状态；
- 顾问确认后锁定，项目运行中不自动升级；
- 无匹配版本时阻塞并提示导入或联系维护人员。

#### FR-05 准则拆解与统一映射

- 一个原始条款可以拆为多个可检查要求；
- 多个可检查要求可以映射到一个统一披露要求；
- 映射类型包括 `equivalent`、`overlapping`、`broader_than`、`narrower_than`、`unique`；
- 映射保留来源、审核状态和差异说明；
- 合并不得删除准则特有的范围、时间、限定条件和定量要求；
- 已审核映射优先；模型新映射必须标记 `unreviewed`。

### 7.4 目录、母版与适配

#### FR-07 动态目录

- 根据准则并集、客户规格、证据覆盖和参考样稿生成目录；
- 展示章节目标、预计篇幅、对应要求、证据覆盖和缺口；
- 顾问可修改标题、顺序、篇幅和颗粒度；
- 客户要求与准则冲突时必须显式提示。

#### FR-08 母版粗稿

- 母版覆盖项目锁定准则要求的并集；
- 事实只能使用客户证据；
- 每个内容块关联章节、统一披露要求和证据；
- 证据不足时按项目设置生成缺口或建议文本；
- 输出达到可直接进入顾问专业审阅的粗稿水平。

#### FR-13 准则适配

- 适配仅针对项目已锁定准则；
- 输出适配稿和差异清单；
- 差异动作包括 `keep`、`condense`、`reorganize`、`supplement`、`omit`；
- 公共内容关联母版内容块，不创建脱离证据库的新事实副本。

### 7.5 评价、缺口与同行参考

#### FR-09 准则回应矩阵

- 覆盖所有项目锁定准则的可检查要求；
- 每条要求必须有五状态之一；
- 不适用必须有人工确认理由；
- 低置信度必须说明原因；
- 可从矩阵定位到母版章节和证据。

#### FR-10 双轨评价

- 准则回应依据锁定准则和客户证据；
- 同业评价不得改变准则回应状态；
- 同业样本由顾问指定，或由 Agent 推荐后经顾问确认；
- 只评价披露覆盖、深度、量化程度和可读性，不评价真实 ESG 绩效。

#### FR-11 缺口与补件

每个缺口包含对应要求、缺失信息、需要原因、建议问题、建议责任部门、优先级和可选建议文本。

项目支持三种模式：`questionnaire` 生成补件问题，`marked_draft` 生成带标记建议文本，`gap_only` 仅记录缺口。

#### FR-12 同行参考报告

- 同行材料标记为 `peer_reference`，不得进入客户证据库；
- 参考方式为 `style_reference`、`quality_benchmark` 或 `both`；
- 可提取目录、章节比例、叙事风格、表格和案例方式；
- 不得长段复制同行正文；
- 联网候选必须经顾问确认后使用。

### 7.6 人工确认、导出与跨 Agent

#### FR-14 人工确认与导出拦截

- `inference` 或 `suggested_text` 转为可对外内容时，必须关联证据或填写人工确认说明；
- `information_gap` 只有补充证据后才能转为事实；
- 内部版保留内容状态和证据引用；
- 干净版存在未确认内容时阻止导出，并返回内容块、章节和原因。

#### FR-15 跨 Agent 复用

- 核心状态存储在项目目录，不存储于专有会话；
- 所有 Agent 适配层调用相同脚本和数据契约；
- 第二个 Agent 能读取当前状态并保留人工修改；
- 工具能力不足时必须明确降级，不重建不兼容结果。

### 7.7 P1 功能规则

#### FR-16 基础数据校验

- 检查同一指标在正文和表格中的年份、单位和数值是否一致；
- 对可识别的简单合计、同比和占比进行复算；
- 无法复算时标记原因，不修改客户原始数据。

#### FR-17 篇幅与颗粒度控制

- 支持报告总字数目标及章节级字数预算；
- 支持 `concise`、`standard`、`detailed` 和自定义章节深度；
- 内容不足时不得通过重复表达强行满足篇幅。

#### FR-18 既有报告保留

- 默认保留原目录、标题和文风，优先局部补强；
- 新增、改写和删除建议必须可识别；
- 重大重构必须先说明原因并获得顾问确认。

#### FR-19 表格与图表建议

- 对结构化披露生成规范表格草案；
- 图表建议必须列出用途、图表类型、所需字段和数据缺口；
- MVP 不承诺完成客户品牌化视觉设计。

#### FR-20 英文版与一致性校核

- 中文稿确认后才询问是否生成英文版；
- 英文版从已确认中文稿生成，不与中文稿并行独立起草；
- 检查中英文术语、数值、单位、年份、范围和目标表述一致性。

## 8. 项目目录与配置契约

### 8.1 标准项目目录

实现可以增加内部文件，但不得改变以下公共路径的含义：

```text
project/
├── project.yaml
├── brief.md                    # 已确认项目规格
├── sources/
│   ├── client/               # 客户事实材料
│   ├── peer/                 # 同行参考材料
│   └── requirements/         # 项目级自定义要求
├── state/
│   ├── workflow.json
│   ├── source_manifest.jsonl
│   ├── evidence.jsonl
│   ├── disclosure_ledger.jsonl    # 要求、证据、正文和评价关系的唯一真相源
│   ├── outline.md
│   └── handoff.json               # 可选；跨 Agent 交接完整性快照
├── drafts/
│   ├── master/
│   └── adaptations/
├── outputs/
│   ├── internal/
│   └── clean/
└── logs/
    ├── harness.jsonl
    ├── trial_metrics.jsonl
    ├── trial_summary.json         # 有试用记录后生成
    └── trial_summary.md
```

### 8.2 `project.yaml` 必填结构

```yaml
project_id: string
project_name: string
client_name: string
reporting_period_start: YYYY-MM-DD
reporting_period_end: YYYY-MM-DD
report_type: sustainability | climate | combined
primary_language: zh-CN
target_length_words: integer | null
granularity: concise | standard | detailed | custom

  - standard_id: string
    version_id: string
    confirmed_by_user: true
custom_requirements: []
gap_handling: questionnaire | marked_draft | gap_only
peer_reference_mode: style_reference | quality_benchmark | both | none
data_policy:
  cloud_processing_allowed: boolean
  web_search_allowed: boolean
  anonymization_required: boolean
  retention: keep | delete_after_export | ask

  master_report: true
  response_matrix: true
  gap_list: true
  evidence_list: true

```

所有配置变化必须通过结构校验。顾问已确认的准则版本、数据授权和人工正文不得被静默覆盖。

Markdown 主交付、回应矩阵、适配稿以及可选 Word/Excel 均从 `disclosure_ledger.jsonl` 派生，不得各自维护独立且不可回写的业务判断。

`handoff.json` 只能保存公共契约哈希、来源复用指纹、工作流、Checkpoint 和下一动作，不能复制或覆盖业务判断。项目公共契约变化后旧交接必须失效。`trial_metrics.jsonl` 为追加式试用事实记录，汇总文件必须绑定源记录哈希。

## 10. 输出规格

### 10.1 母版报告

- 中文 Markdown 文档，文件名为 `outputs/markdown/master_report.md`；
- 结构与篇幅遵循已确认项目规格；
- 使用 `[待确认-推断]`、`[建议文本]`、`[信息缺口]` 行内标记，并在 HTML 注释中记录内容编号和证据编号；
- 章节必须可关联到统一披露要求和内容块。

### 10.2 准则回应矩阵 `response_matrix.xlsx`

固定字段顺序：准则名称、准则版本、原始条款编号、可检查要求编号、可检查要求、统一披露要求编号、回应状态、母版章节、回应摘要、证据编号、证据位置、判断理由、缺失信息、改进建议、置信度、置信度原因、人工审阅状态、人工备注。

### 10.3 缺口清单 `gap_list.xlsx`

固定字段顺序：要求编号、准则与条款、缺失信息、需要原因、建议补件问题、建议责任部门、优先级、建议文本、处理状态、人工备注。

### 10.4 证据清单 `evidence_list.xlsx`

固定字段顺序：证据编号、文件、页码或表格位置、证据摘录、期间、单位、关联统一披露要求、关联准则条款、证据关系、解析状态、人工备注。

证据清单仅供咨询团队内部使用。

### 10.5 适配稿与差异清单

每个适配任务输出：

- `outputs/markdown/adapted_<standard_id>.md`；
- `outputs/markdown/report_manifest.json` 中的目标准则和文件哈希；
- 可选的现有内部 DOCX/XLSX 派生文件，不作为 Markdown-first MVP 的完成门槛。

差异清单字段：母版内容编号、目标准则、动作、适配原因、适配后章节、补充证据、内容状态、人工审阅状态。

## 11. Agent 交互契约

### 11.1 必须询问用户

- 数据处理授权；
- 客户材料、既有报告或模板、项目规格确认；
- 顾问指定准则后的版本确认；
- 报告目录；
- 关键证据缺口的处理方式；
- 同行样本及参考方式；
- 优秀案例已提供或明确不提供，以及提供时用于风格、质量或二者；
- 干净版导出前未确认内容的处理。

### 11.2 不得自行决定

- 客户适用哪套准则；
- 将未经审核的规则解释当成正式规则；
- 将项目准则自动升级到新版本；
- 将同行事实写成客户事实；
- 将推断或建议文本当作已确认事实；
- 静默删除客户要求或准则特有要求；
- 静默覆盖人工编辑内容。

### 11.3 可自动执行

- 扫描和分类文件；
- 对未变化文件复用解析结果；
- 加载已确认规则版本；
- 生成项目规格、目录、母版、矩阵和清单草案；
- 执行结构校验、覆盖检查和导出前检查；
- 对失败文件生成错误记录并继续处理其他文件。

## 12. 模块边界

| 模块 | 负责 | 不负责 |
| --- | --- | --- |
| Agent Adapter | 触发、提问、展示进度、调用核心命令 | 保存唯一业务状态、解析文件 |
| Project Manager | 项目初始化、配置校验、状态流转 | 语义判断和正文生成 |
| Source Ingestion | 文件哈希、解析、定位、证据结构化 | 判断是否满足准则 |
| Standards Registry | 规则版本、条款、审核状态 | 决定客户适用准则 |
| Mapping Engine | 条款拆解、统一披露映射、差异关系 | 生成对外最终结论 |
| Evidence Engine | 检索、关联、冲突提示、证据复用 | 创造客户事实 |
| Drafting Engine | 目录、母版、补件文本和适配稿 | 绕过内容状态和导出门禁 |
| Assessment Engine | 五状态评价、理由和置信度 | 将同业水平混入准则结论 |
| Exporter | Markdown 主交付、哈希清单及可选 Word/Excel 派生输出 | 修改业务判断 |

确定性工作优先由脚本完成，包括文件哈希、解析位置、模式校验、状态流转、公式检查和导出。语义拆解、检索排序、映射建议、正文起草和评价理由由模型完成，并保留人工审阅入口。

## 13. 异常与降级处理

| 场景 | 系统行为 |
| --- | --- |
| 文件损坏或受密码保护 | 标记失败文件和原因，继续其他文件；关键文件失败时阻塞 |
| 扫描 PDF 无可提取文本 | 标记需要 OCR，MVP 不假装解析成功 |
| Excel 公式无法计算 | 保留原公式和显示值，标记未复算 |
| 准则版本未找到 | 阻塞准则锁定，提示导入或联系维护人员 |
| 规则映射未经审核 | 标记待确认，不显示为正式映射 |
| 证据互相矛盾 | 回应状态设为待确认，列出冲突证据 |
| 资料不足 | 按项目设置生成补件、建议文本或缺口 |
| 模型调用失败 | 保留已完成状态，可重试当前步骤，不重跑已完成解析 |
| Agent 缺少某项工具 | 明确降级范围，复用已有状态，不另建不兼容结果 |
| 干净版包含未确认内容 | 阻止导出并列出内容块、章节和原因 |

## 14. 非功能需求

### 14.1 本地与安全

- 默认项目目录本地存储；
- 外部模型和联网权限按项目确认；
- 日志不得记录未脱敏的完整客户原文；
- 支持按用户要求清理中间文件；
- 客户材料不得进入规则库或其他客户项目。

### 14.2 可移植性

- 公共项目文件使用 YAML、JSON/JSONL 和 Markdown；DOCX、XLSX 为输入或可选派生输出；
- 核心状态不得只存在于模型会话；
- Agent 适配层不得修改公共数据语义；
- MVP 通过统一 Skill、公共状态契约和独立进程测试验证可移植性；产品特定 Agent 的真实外部调用不作为完成条件。

### 14.3 可恢复性

- 每个长步骤完成后写入状态；
- 失败后可从最近完成步骤继续；
- 已解析且哈希未变化的文件不得重复处理；
- 所有已保存的人工修改必须可恢复。

### 14.4 可解释性

- 所有评价包含规则、正文、证据和理由；
- 低置信度必须说明原因；
- 规则冲突、证据冲突和口径不一致必须可见；
- 不以单一总分替代逐条判断。

### 14.5 可用性

- 普通顾问无需编写命令或手工编辑配置；
- Agent 使用业务语言提问，一次只确认真正影响结果的事项；
- 高级用户可以查看和修改项目配置及中间结果；
- 长任务必须显示当前阶段、已完成步骤和下一步所需输入。

## 15. 试用指标

MVP 不设置固定节省时长，试用时记录：

- 人工基线总工时；
- Agent 运行时间；
- 顾问审阅和修正时间；
- 重复解析或重复撰写次数；
- 条款拆解与映射纠错数；
- 证据位置错误数；
- 母版可直接保留、局部修改和需要重写的内容比例；
- 所选准则条款覆盖率；
- 跨进程或跨工具接续时重复处理的文件数量。

试用结论必须同时报告节省工时和人工纠错情况，不能只报告生成速度。

## 16. 验收用例

`AC-01` 至 `AC-12` 继承自 `REQUIREMENTS.md`，是稳定业务验收基线；`AC-13` 至 `AC-19` 为既有扩展用例；新增用例只能从 `AC-20` 以后追加。禁止复用、交换或重编已发布编号。

| 编号 | 前置条件 | 操作 | 预期结果 |
| --- | --- | --- | --- |
| AC-01 | 空项目且未确认授权和规格 | 用户通过自然语言启动项目 | 完成数据授权与项目规格确认，并生成可持久化项目配置 |
| AC-02 | 已确认报告期间 | 顾问选择多套准则 | 系统推荐版本，顾问确认后锁定 |
| AC-03 | 两套准则存在相同或相近要求 | 执行统一映射 | 合并为统一披露要求，并保留原条款、映射类型和差异 |
| AC-04 | 同一客户证据支持多个要求 | 执行证据映射 | 证据只解析一次，可关联多个统一披露要求和准则条款 |
| AC-05 | 正式目录和 Anchor 已确认 | 生成完整母版 | 生成覆盖所选准则并集的中文母版粗稿 |
| AC-06 | 母版包含事实、推断、建议和缺口 | 执行内容检查 | 四类内容得到明确区分，状态可持久化和追溯 |
| AC-07 | 母版和准则并集已生成 | 生成回应矩阵 | 逐条输出五状态、理由、证据位置和置信度 |
| AC-08 | 顾问已确认同行样本 | 执行双轨评价 | 同业评价独立输出，不改变准则回应状态 |
| AC-09 | 母版已完成 | 顾问选择目标准则 | 输出适配稿和差异清单，公共内容仍关联母版 |
| AC-10 | 干净版包含未确认内容 | 尝试导出 | 导出被阻止，并返回具体内容块、章节和原因 |
| AC-11 | 一个受支持 Agent 已保存项目 | 第二个 Agent 打开同一目录 | 读取相同配置、账本和状态并继续任务 |
| AC-12 | 端到端试用完成 | 汇总试用结果 | 记录人工基线、AI 时间、人工修正时间和映射纠错情况 |
| AC-13 | 一个 Agent 已完成解析 | 第二个 Agent 打开同一项目 | 可继续流程，不重新解析未变化文件 |
| AC-14 | 人工已修改正文 | 再次运行评价 | 保留人工文本并基于最新内容更新评价 |
| AC-15 | 中文稿已确认 | 流程继续 | Agent 询问是否生成英文版，不自动翻译 |
| AC-16 | 正文和表格包含同一指标 | 执行基础数据校验 | 发现年份、单位或数值不一致并定位问题 |
| AC-17 | 项目设置章节字数预算 | 生成母版 | 输出遵循预算且不以重复文字填充 |
| AC-18 | 已有报告作为初稿 | 执行补强 | 保留原结构，修改可识别，重大重构先确认 |
| AC-19 | 中文稿已确认并选择翻译 | 生成英文版 | 术语、数值、单位、年份和范围与中文稿一致 |
| AC-20 | 证据库和准则并集已生成 | 尝试直接生成目录 | 状态停留在证据确认，直到关键缺口、映射和冲突证据被确认 |
| AC-21 | Evidence Checkpoint 已通过 | 生成正式目录 | 展示篇幅、要求、证据覆盖和预计缺口，等待目录确认 |
| AC-22 | 正式目录已确认 | 生成 Anchor | 只生成代表性章节及对应矩阵，并进入 Anchor 确认状态 |
| AC-23 | Anchor 被要求修改 | 尝试生成完整母版 | 系统拒绝跳过 Checkpoint，保留修改意见和当前状态 |
| AC-24 | Anchor 已确认 | 生成完整母版 | 按已确认颗粒度、文风、标记和评价方式生成其余章节 |
| AC-25 | 项目停在 Anchor 或导出确认状态 | 重启并重新打开项目 | 恢复流程状态、Checkpoint 状态、关联产物和人工备注 |
| AC-26 | 含推断和建议文本 | 导出内部版 | 导出成功并保留状态标记 |
| AC-27 | 新项目尚未确认输入 | Agent 准备生成报告 | 询问并持久化客户材料、既有报告或模板、目标准则、优秀案例选择及报告偏好 |
| AC-28 | 母版与全部准则适配方案已具备 | 生成 Markdown 交付 | 输出一份联合母版和每个已确认准则各一份适配稿 |
| AC-29 | Markdown 含非确认内容和客户事实 | 检查交付文件 | 保留三类风险标记、内容编号和证据编号 |
| AC-30 | Markdown 已生成 | 修改上游账本或输出文件后校验 | 哈希清单判定结果过期并指出问题 |
| AC-31 | Codex 已保存同一项目 | Claude Code 读取并继续 | 使用相同项目状态和 Skill 契约完成真实接续；WorkBuddy、Trae 仅提供通用说明 |
| AC-32 | 通用 Skill 与项目状态已生成 | 为 Codex、Claude Code、WorkBuddy 或 Trae 配置接入 | 各工具读取同一 Skill、状态和命令，不复制业务判断；未实测产品不被宣称为已支持 |

`AC-31` 的编号与原语义保留，但按用户 2026-07-17 的范围决定取消，不纳入当前 MVP 完成门槛。通用多工具可移植性改由 `AC-32` 验收。

## 17. 开发阶段与交付门槛

### 17.1 阶段 0：测试夹具和规则样例

**交付**：脱敏项目材料、至少两套准则的已审核拆解与交叉映射样例、人工期望矩阵和母版片段。

**门槛**：专业人员确认样例可作为开发和测试基准。

### 17.2 阶段 1：项目骨架与数据契约

**交付**：项目初始化、`project.yaml` 校验、状态机、公共数据模型、日志和错误格式。

**门槛**：可创建项目、恢复状态，并通过数据结构单元测试。

### 17.3 阶段 2：资料解析与证据库

**交付**：Word、PDF、Excel 解析器，文件哈希，证据定位与复用。

**门槛**：对测试材料准确定位正文和表格；重复运行不重复解析。

### 17.4 阶段 3：准则库与映射

**交付**：规则版本加载、准则锁定、条款拆解、统一披露映射和审核状态。

**门槛**：测试准则形成完整并集，所有原条款均可追溯且无静默丢失。

### 17.5 阶段 4：母版、评价和清单

**交付**：动态目录、母版粗稿、回应矩阵、缺口清单、证据清单和导出门禁。

**门槛**：通过 `AC-05` 至 `AC-08`、`AC-20` 至 `AC-24` 及 `AC-26`，顾问完成一次专业审阅。

### 17.6 阶段 5：适配、Agent 接入与端到端试用

**交付**：Markdown 母版、逐准则适配稿、通用 Skill/状态契约、多工具安装说明和试用指标记录。

**门槛**：完成 `AC-09`、`AC-10`、`AC-12` 至 `AC-14`、`AC-25` 至 `AC-30` 及 `AC-32`；`AC-11`、`AC-15` 至 `AC-19`、`AC-31` 属于后续真实多 Agent、英文、数据校验、篇幅和 Word 增强，不阻塞 Markdown-first MVP。

## 19. Definition of Done

MVP 仅在同时满足以下条件时视为完成：

- P0 功能全部实现并通过对应验收用例；
- 至少两套准则完成已审核的拆解和统一映射测试；
- 一套脱敏或模拟项目完成全流程；
- 母版、回应矩阵、缺口清单、证据清单和一个准则适配稿均可生成；
- 未确认内容无法进入干净版；
- 通用 Skill 和公共状态不绑定单一 Agent，独立进程可读取相同状态并继续任务；
- 顾问完成专业审阅并记录映射、证据和正文纠错情况；
- 已知限制、失败文件和未审核映射均明确显示；
- 团队完成试用复盘并决定下一阶段范围。
