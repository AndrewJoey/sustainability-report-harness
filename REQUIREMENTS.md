# 多准则可持续披露报告 AI 能力包需求文档

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 文档状态 | V1.1，待内部评审 |
| 产品阶段 | 可试用 MVP |
| 目标用户 | 可持续发展、ESG、气候披露咨询团队 |
| 使用方式 | 在不同 AI Agent 中以自然语言调用的本地能力包 |
| 默认语言 | 中文；中文稿确认后询问是否生成英文版 |

## 2. 一页式决策摘要

### 2.1 要解决的问题

顾问当前需要反复拆解不同准则、寻找相同要求、搜索客户证据、撰写相似内容并逐条检查回应情况。客户对报告篇幅、颗粒度、结构和参考样稿的要求又各不相同，导致报告初稿通常需要从头组织。

### 2.2 MVP 方案

构建一个咨询团队内部使用的本地 AI 能力包。顾问指定本项目准则，系统将不同准则合并为统一披露要求，复用同一证据库生成覆盖所选准则并集的全面母版粗稿，再按具体准则和客户要求派生适配版本。

### 2.3 用户获得什么

- 一份覆盖所选准则并集、可直接进入专业审阅的中文母版粗稿；
- 一份逐条说明回应状态、对应章节、证据、理由和置信度的准则回应矩阵；
- 一份缺口与补件清单以及内部证据引用清单；
- 按指定准则从母版派生的适配粗稿；
- 一套可跨 Agent 继续使用的项目配置和中间结果。

### 2.4 最关键的产品判断

- 适用准则由顾问指定；系统仅推荐版本，并必须由顾问确认。
- 母版覆盖本项目所选准则要求的并集，不覆盖规则库中的全部准则。
- “准则回应程度”与“同业/最佳实践水平”分开评价。
- 事实必须有客户证据；推断和建议文本须明确标记并经人工确认。
- 准则映射准确性和证据可追溯性优先于文字精致度和运行速度。
- MVP 是可复用工程能力包，不是 SaaS，也不是单一 Prompt。

### 2.5 本轮明确不做

年度滚动、完整碳核算、重大性分析、气候情景分析、复杂协作审批、完整视觉设计和无需人工审阅的对外终稿均不属于本次 MVP。

## 3. 背景与问题

咨询团队在编制可持续披露报告或气候披露报告时，需要同时理解和回应港交所、上交所、深交所、北交所及客户指定的其他准则或要求。不同准则之间存在大量相同、相近但细节不完全一致的披露要求，当前工作通常需要顾问重复完成以下步骤：

1. 阅读并拆解不同准则；
2. 识别相同、相近及特有要求；
3. 在客户资料中反复寻找证据；
4. 针对不同准则重复撰写或调整内容；
5. 检查报告是否逐条回应准则；
6. 根据不同客户的篇幅、颗粒度、模板和参考样稿重建初稿。

本产品希望将这些工作沉淀为可复用能力，使顾问只需整理一次客户材料、撰写一次核心内容，即可形成覆盖所选准则并集的全面母版粗稿，并按不同准则快速派生适配版本。

## 4. 产品定位

### 4.1 产品定义

一个面向咨询团队内部使用、本地优先、可在不同 AI Agent 中复用的可持续披露报告能力包。

产品不是独立 SaaS，也不是单一提示词。其最终形态包括：

- 框架无关的核心工作流和数据契约；
- 经专业人员审核的结构化准则库；
- 文档解析、证据索引、映射、校验和导出脚本；
- 面向 Codex、Claude Code、Gemini CLI、Cursor 等 Agent 的轻量适配层；
- 可复跑、可迁移的项目配置和中间结果；
- 面向顾问的自然语言交互方式。

### 4.2 核心价值

1. **一次拆解，多准则复用**：将不同准则映射到统一披露要求，减少逐套准则重复分析。
2. **一次取证，多处回应**：客户资料只解析一次，同一证据可关联多个披露要求和准则条款。
3. **一次成稿，多版适配**：先生成全面母版粗稿，再按具体准则及客户要求调整。
4. **判断可解释**：每项回应结论均展示依据、证据位置、评价理由和置信度。
5. **顾问保留控制权**：适用准则、版本、报告规格、参考样稿和最终内容均由顾问确认。

### 4.3 产品原则

- 准则映射准确性和证据可追溯性优先于文字润色和生成速度。
- AI 输出均为内部专业辅助意见，不构成法律、审计、鉴证或监管结论。
- 事实不得脱离客户证据生成；建议可以生成，但必须明确标记并经人工确认。
- 不重复解析已有材料，不重复撰写已确认内容，不重复执行无变化的检查。
- 规则、证据、内容和 Agent 指令相互分离，避免绑定单一模型或 Agent。

## 5. 目标与非目标

### 5.1 MVP 目标

- 跑通“项目规格确认－资料解析－多准则合并－母版粗稿－回应矩阵－缺口清单－准则适配”的完整流程。
- 支持顾问指定港交所和三大交易所准则，并支持加入 ISSB、GRI、客户模板、评分问卷等项目级要求。
- 支持从零散客户材料生成报告，也支持对已有报告进行评估和局部补强。
- 根据不同客户的篇幅、颗粒度、目录、文风和参考样稿生成可审阅粗稿。
- 记录人工基线、AI 运行时间、审阅修正时间和可复用比例，用真实试用结果衡量效率提升。

### 5.2 MVP 非目标

- 不提供正式合规认证或替代专业人员作出适用性判断。
- 不构建完整的碳核算、双重重大性分析或气候情景分析引擎。
- 不追求无需人工审阅即可直接交付甲方的终稿。
- 不建设完整 SaaS、复杂权限、多级审批或多人实时协作平台。
- 暂不支持同一客户的年度滚动更新流程。
- 暂不承担报告完整视觉设计，仅生成规范表格和图表建议。

## 6. 用户与使用场景

### 6.1 目标用户

- 报告撰写顾问：整理资料、生成初稿、补充披露内容。
- 项目经理或审阅人员：检查准则回应、证据充分性和修改优先级。
- 规则库专业维护人员：审核准则拆解、映射关系和版本更新。

### 6.2 核心场景

#### 场景 A：从零散资料生成母版粗稿

顾问向 Agent 提供历史报告、制度、数据表、访谈纪要等文件，指定准则和客户要求。Agent 建立项目规格与证据库，生成覆盖所选准则并集的全面母版粗稿，并输出回应矩阵和补件清单。

#### 场景 B：评估并补强已有报告

顾问提供客户已有报告或初稿。Agent 在尽量保留原结构和文风的前提下，判断各准则要求的回应状态，标记缺口并生成可用的补充文本。重大结构调整须先征求顾问确认。

#### 场景 C：按具体准则派生适配版本

顾问从母版选择目标准则。Agent 调整目录、篇幅、颗粒度、措辞和特有披露内容，输出该准则的适配粗稿，同时保留与母版及证据库的关联。

#### 场景 D：使用同行报告作为参考

顾问可以自行提供同行报告，也可以要求 Agent 联网推荐候选。Agent 先询问顾问是否有关注企业；联网候选必须由顾问确认后使用，样本数量按项目决定。

同行报告可用于两种目的：

- 风格复刻：借鉴目录、章节比例、叙事方式、表格和案例呈现；
- 质量标杆：比较披露覆盖、深度、量化程度和可读性。

同行报告不得作为客户事实证据，也不得直接复制其正文。

## 7. 端到端工作流程

### 7.1 项目启动与数据确认

1. 用户将 Word、PDF、Excel 等客户材料交给 Agent。
2. Agent 在执行前询问数据处理边界，包括是否允许云端模型、联网搜索、脱敏及保留中间文件。
3. 默认项目目录本地优先；联网搜索同行资料时不得发送客户内部材料。

### 7.2 项目规格确认

Agent 从用户指令、客户任务书、模板和参考样稿中提取并生成项目规格，至少包括：

- 客户主体与报告期间；
- 报告类型、用途及目标读者；
- 顾问指定的准则；
- 目标篇幅、整体及章节颗粒度；
- 客户指定目录、模板和必须覆盖议题；
- 文风、品牌用语和禁用表达；
- 表格、案例、指标和图表需求；
- 参考样稿的使用方式；
- 缺失信息的处理方式；
- 预期交付物。

项目规格必须由顾问确认后生效。普通用户只需自然语言交流；高级用户可查看和修改配置文件。

### 7.3 准则与版本确认

- 适用准则完全由顾问指定，Agent 不自行作出适用性判断。
- 系统根据报告期间推荐对应准则版本，展示生效日期和版本说明。
- 顾问确认后锁定项目规则集合；执行中不得自动切换版本。
- 客户要求和临时导入的规则只在项目内生效，并与正式准则清晰区分。

### 7.4 资料解析与证据库

- MVP 优先稳定支持 Word、PDF、Excel。
- 将资料拆分为可检索证据片段，并保留文件名、页码、工作表和单元格区域。
- 识别年份、单位、组织边界、指标口径及表文关系。
- 相同资料只解析一次；后续准则映射和报告生成直接复用证据库。
- OCR 扫描件、PPT、邮件和网页等作为后续增强能力。

### 7.5 统一披露要求与准则并集

- 将各准则条款拆解为可检查的披露要求。
- 识别相同、相近、包含及特有要求，形成统一披露要求库。
- 本项目母版仅覆盖顾问已选择准则的要求并集，不纳入无关准则。
- 保留统一披露要求与各准则原始条款之间的多对多关系。
- 客户模板与准则底线冲突时，Agent 必须明确提示，不得静默牺牲准则要求。

### 7.6 目录与母版生成

- 综合准则并集、客户要求、重要性分析结果、现有资料和参考样稿动态生成目录。
- 给出章节目标、预计篇幅、对应准则和证据覆盖情况。
- 顾问确认目录后生成中文母版粗稿。
- 母版为内部“披露内容母稿”，可比最终客户版本更全面，并保留候选内容与适配标签。
- 第一版应达到：结构完整、证据支持内容基本成文、缺口和建议清晰标记、可直接进入顾问专业审阅。

### 7.7 评估、补件与适配

- 生成逐条准则回应矩阵及双轨评价。
- 对未回应或证据不足项，按用户项目设置生成补件清单、带标记建议文本或仅标注缺口。
- 针对具体准则，从母版选择、压缩、重组并补充特有内容，生成适配粗稿。
- 未变化内容直接复用，不重复取证和重写。

### 7.8 人工确认节点

长项目默认设置四个确认节点：

1. 项目规格；
2. 报告目录；
3. 关键证据缺口；
4. 母版及适配成稿。

## 8. 内容真实性与证据规则

AI 生成内容分为以下四类：

| 类型 | 定义 | 是否可直接进入正式报告 |
| --- | --- | --- |
| 已确认事实 | 有明确客户资料支持，并可定位来源 | 可以 |
| 合理推断 | 可由多份材料间接推导，但原文未明确说明 | 人工确认后可以 |
| 建议文本 | 依据准则、同行或最佳实践生成的候选内容 | 人工确认且补充事实后可以 |
| 信息缺口 | 缺少关键事实、指标、治理机制或证明材料 | 不可以 |

正式导出前必须检查并拦截未经确认的推断、建议文本、占位符和无证据事实。

证据引用仅用于咨询团队内部，不显示在面向甲方的干净报告中。内部矩阵须保留文件、页码、工作表或单元格位置；必要时可另行生成带批注的内部审阅版。

## 9. 评价体系

### 9.1 准则回应评价

每条要求采用五种状态：

- 充分回应；
- 部分回应；
- 未回应；
- 不适用；
- 待确认。

判断维度至少包括完整性、相关性、证据充分性、定量信息、口径一致性和可追溯性。每项结论须包含：

- 状态；
- 判断理由；
- 对应报告章节；
- 证据位置；
- 缺失内容；
- 改进建议；
- 高、中或低置信度及其原因。

### 9.2 同业与最佳实践评价

同业评价与准则回应评价分开，不得用同行普遍做法替代准则要求。

同业评价采用“领先、同等、落后”，重点比较：

- 议题和要求覆盖程度；
- 披露深度；
- 指标、目标和量化信息；
- 内容组织和可读性。

MVP 不评价企业实际 ESG 绩效，也不形成正式行业排名。对标企业由 Agent 推荐并经顾问确认，或由顾问直接指定；样本数量由顾问按项目决定。

## 10. 功能需求

### 10.1 P0：MVP 必须具备

| 编号 | 功能 | 最低可接受结果 |
| --- | --- | --- |
| FR-01 | 自然语言启动与项目规格确认 | Agent 从用户说明和材料中形成项目规格，顾问确认后才能继续 |
| FR-02 | 数据处理授权确认 | 每个项目单独确认云端处理、联网、脱敏和中间文件保留方式 |
| FR-03 | Word、PDF、Excel 资料解析 | 可提取正文和表格，并定位到文件、页码、工作表或单元格区域 |
| FR-04 | 准则及版本确认 | 顾问指定准则；系统按报告期间推荐版本并等待确认 |
| FR-05 | 多准则统一映射 | 条款可拆解、去重、合并并保留与原始条款的多对多关系 |
| FR-06 | 证据复用 | 同一证据解析一次即可回应多个统一披露要求和准则条款 |
| FR-07 | 动态目录 | 根据准则并集、客户规格、已有资料和参考样稿生成可确认目录 |
| FR-08 | 中文母版粗稿 | 生成覆盖所选准则并集、结构完整且可审阅的全面粗稿 |
| FR-09 | 准则回应矩阵 | 逐条输出五状态、理由、章节、证据、缺口、建议和置信度 |
| FR-10 | 双轨评价 | 准则回应程度与同业/最佳实践水平分开输出 |
| FR-11 | 缺口与补件 | 对证据不足项生成缺口、补件问题或带标记建议文本 |
| FR-12 | 同行报告分析 | 区分风格复刻与质量标杆，不将同行内容作为客户事实 |
| FR-13 | 准则适配粗稿 | 从母版选择、压缩、重组并补充特有内容，不重复撰写共有内容 |
| FR-14 | 内容状态与导出拦截 | 区分事实、推断、建议和缺口，阻止未确认内容进入干净终稿 |
| FR-15 | 跨 Agent 复用 | 项目配置、规则、证据和中间结果不依赖单一 Agent 会话 |

### 10.2 P1：MVP 应当具备

| 编号 | 功能 | 最低可接受结果 |
| --- | --- | --- |
| FR-16 | 基础数据校验 | 检查年份、单位、表文一致性及可复算的简单公式 |
| FR-17 | 篇幅与颗粒度控制 | 支持报告总篇幅和章节级深度、字数预算 |
| FR-18 | 既有结构保留 | 修改已有报告时优先局部补强，重大重构前询问顾问 |
| FR-19 | 表格与图表建议 | 生成规范披露表，并说明建议图表及所需数据 |
| FR-20 | 英文版询问与校核 | 中文稿确认后询问是否翻译，并检查术语、指标和口径一致性 |

### 10.3 P2：后续能力

- 年度滚动更新；
- OCR、PPT、邮件、网页和压缩包处理；
- 多人协作、任务分派和审批；
- 完整报告视觉设计；
- 基础碳核算、重大性分析和气候风险分析工具；
- MCP 或本地服务接口；
- 行业统计和大样本同业分析。

## 11. 标准输入与输出

### 11.1 项目输入

- 项目规格；
- 顾问选择并确认版本的准则集合；
- 客户报告、制度、数据台账和访谈材料；
- 客户模板、任务书和写作要求；
- 客户或顾问提供的同行参考报告；
- 用户确认的数据处理边界。

### 11.2 默认业务交付物

| 交付物 | 用途 | 默认格式 |
| --- | --- | --- |
| 母版报告 | 覆盖所选准则并集的全面中文粗稿 | Word |
| 准则回应矩阵 | 逐条映射、回应状态、理由、证据和置信度 | Excel |
| 缺口与补件清单 | 缺失信息、建议问题、优先级和建议责任部门 | Excel |
| 证据引用清单 | 内部证据追溯 | Excel |
| 准则适配版 | 从母版派生的特定准则粗稿 | Word |

中文母版确认后，Agent 询问用户是否生成英文版。英文版采用“先定稿一种语言，再翻译和一致性校核”的方式，不默认同步生成。

## 12. 核心数据对象

为支持跨 Agent 复用，MVP 至少定义以下框架无关的数据对象：

- `ProjectSpec`：项目、客户、期间、语言、篇幅、颗粒度和输出要求；
- `StandardVersion`：准则名称、版本、生效日期、来源和审核状态；
- `Requirement`：原始条款及拆解后的可检查要求；
- `UnifiedDisclosure`：合并后的统一披露要求及其准则映射；
- `Evidence`：证据片段、来源位置、年份、单位和适用范围；
- `DisclosureContent`：母版中的披露内容、内容类型和人工确认状态；
- `Assessment`：五状态评价、理由、置信度、缺口和改进建议；
- `PeerBenchmark`：同业样本、比较维度和相对评价；
- `Adaptation`：特定准则版本相对母版的保留、压缩、补充和重组关系。

具体文件格式可在技术设计阶段确定，但必须可读、可版本控制，且不依赖某一 Agent 的私有会话状态。

## 13. 规则库治理

- 正式规则库优先收录交易所正式规则及官方指引，并记录来源、版本和生效日期。
- AI 负责初步解析条款、提出拆解和新旧版本差异。
- 内部专业人员负责审核、修正和批准。
- 审核通过后发布为正式、不可静默修改的规则版本。
- 普通顾问直接使用已发布版本；项目级自定义要求不得覆盖或伪装成正式监管规则。
- 准则更新时保留旧版本，项目不得因规则库更新而自动迁移。

## 14. 非功能需求

### 14.1 可移植性

- 核心规则、项目数据和中间结果不得只存在于 Agent 对话中。
- Agent 适配层应保持轻量，核心能力不依赖单一厂商专有功能。
- 同一项目目录可被不同 Agent 继续处理，不需要重新解析全部材料。

### 14.2 安全与隐私

- 每次项目执行前确认数据处理方式，不做永久默认授权。
- 未经确认，不向外部服务发送客户内部材料。
- 联网检索同行报告时只使用必要的公开筛选信息。
- 支持按用户要求脱敏及删除索引、中间文件。

### 14.3 可解释性

- 所有准则回应结论必须关联规则、报告内容和证据。
- 低置信度结论必须说明原因，不得只给出分数。
- 对规则冲突、资料矛盾和口径不一致应明确提示。

### 14.4 可用性

- 普通顾问无需编写命令或手工编辑配置。
- Agent 应使用业务语言提问，并一次只确认真正影响结果的事项。
- 高级用户可查看和修改配置、规则选择及中间结果。

## 15. MVP 工程交付形态

MVP 最终应是一个本地工程能力包，而非单一 Markdown 或 Prompt。至少包含：

- 通用核心工作流和数据契约；
- 经审核的准则库样例；
- 文档解析、证据索引、映射、校验和导出脚本；
- 多 Agent Skill 或指令适配文件；
- 标准项目目录模板；
- 一套脱敏或模拟的端到端示例；
- 自动化校验及试用记录模板。

需求文档是 MVP 开发的依据，不等同于最终工具。

## 16. MVP 成功指标与验收标准

### 16.1 试用成功指标

MVP 不预设固定节省时长。首轮试用对人工流程和 Agent 流程采用相同任务口径，记录以下指标：

| 指标 | 计算或观察方式 | MVP 判断方向 |
| --- | --- | --- |
| 总工时变化 | 人工基线总工时与 Agent 加人工审阅总工时对比 | 应明显下降 |
| 重复工作减少 | 顾问是否需要重复解析同一资料、重复撰写共有回应 | 原则上不应重复 |
| 映射纠错率 | 顾问修正的条款拆解或映射数占总映射数比例 | 越低越好，所有错误可追溯 |
| 证据有效率 | 引用位置准确且能支持结论的证据占比 | 应满足专业审阅需要 |
| 粗稿可用度 | 顾问可保留、局部修改和需重写的内容比例 | 以可直接进入专业审阅为合格 |
| 覆盖完整性 | 所选准则是否均进入矩阵，未回应项是否被识别 | 不得漏掉已纳入的条款 |
| 跨 Agent 连续性 | 第二个 Agent 能否读取项目状态并继续任务 | 不得重新解析全部材料 |

### 16.2 端到端验收

使用一套真实脱敏或模拟项目材料完成端到端测试，至少满足：

`AC-01` 至 `AC-12` 是仓库验收编号的稳定基线。PRD、测试名称、提交记录和项目计划必须沿用相同语义，不得复用或重编这些编号；新增用例只能从未使用的后续编号追加。

1. `AC-01`：顾问可通过自然语言完成项目启动和项目规格确认，对应 `FR-01`、`FR-02`；
2. `AC-02`：可选择多套准则并确认系统推荐的版本，对应 `FR-04`；
3. `AC-03`：相同或相近条款能合并为统一披露要求，同时保留原条款映射，对应 `FR-05`；
4. `AC-04`：同一客户证据无需重复解析即可支持多个准则，对应 `FR-03`、`FR-06`；
5. `AC-05`：能生成覆盖所选准则要求并集的中文母版粗稿，对应 `FR-07`、`FR-08`；
6. `AC-06`：母版中的事实、推断、建议和缺口得到明确区分，对应 `FR-14`；
7. `AC-07`：回应矩阵能逐条给出五状态、理由、证据位置和置信度，对应 `FR-09`；
8. `AC-08`：同业评价与准则回应评价保持独立，对应 `FR-10`、`FR-12`；
9. `AC-09`：能生成至少一个具体准则的适配粗稿和差异化补充内容，对应 `FR-13`；
10. `AC-10`：未确认推断、建议文本和占位内容无法进入干净终稿，对应 `FR-14`；
11. `AC-11`：另一受支持 Agent 能读取同一项目目录并继续任务，对应 `FR-15`；
12. `AC-12`：试用过程完整记录人工基线、AI 生成时间、人工修正时间和映射纠错情况。

MVP 质量优先级依次为：准则映射准确性、证据可追溯性、内容真实性、覆盖完整性、粗稿可编辑性、文字精致度、运行速度。

## 17. 主要风险与控制措施

| 风险 | 影响 | MVP 控制措施 |
| --- | --- | --- |
| AI 错误拆解或合并准则 | 导致遗漏或错误回应 | 正式规则由专业人员审核；映射保留原文和人工修正入口 |
| 无证据生成客户事实 | 形成失实披露 | 四类内容状态、来源追溯、干净终稿导出拦截 |
| 母版为了覆盖并集而过长 | 难以审阅和适配 | 母版作为内部内容母稿；通过章节预算和适配规则生成客户版本 |
| 同行材料被误作客户事实 | 污染报告内容 | 同行材料独立分类，只用于结构、表达和质量参考 |
| 跨 Agent 表现不一致 | 项目无法稳定复用 | 使用框架无关数据契约；关键解析、校验和导出使用确定性脚本 |
| 客户资料外发 | 违反保密要求 | 每项目确认数据边界；本地优先；联网时不发送客户内部材料 |

## 18. 开发前准备

进入 MVP 开发前需准备：

- 港交所、上交所、深交所和北交所准则的确认版本及官方来源；
- 内部专业人员认可的条款拆解和映射样例；
- 一套脱敏或模拟的客户资料；
- 至少一份客户任务书或报告规格样例；
- 至少一份同行参考报告；
- 一份人工完成的期望母版片段和回应矩阵，用作验证基准；
- 计划首批验证的 Agent 清单及其本地文件和脚本能力说明。

以下事项不阻碍需求定稿，但需在 MVP 开发启动前由团队确认：
