# 多准则可持续披露报告 AI 能力包项目计划

## 1. 文档用途

本文档是项目管理和跨对话交接入口，用于记录当前状态、开发依赖、工作流、里程碑、责任人和下一步行动。

核心文档及职责如下：

- [REQUIREMENTS.md](./REQUIREMENTS.md)：业务需求和内部评审依据；
- [PRD.md](./PRD.md)：开发功能、数据契约、验收用例和完成标准；
- [HARNESS_ARCHITECTURE.md](./HARNESS_ARCHITECTURE.md)：Garden 风格 Skill Harness 架构和稳定性协议；
- `PROJECT_PLAN.md`：项目状态、任务顺序、依赖和交接说明。

文档优先级和约束关系如下：

1. `REQUIREMENTS.md` 定义业务意图、目标和范围；
2. `PRD.md` 将业务意图转化为外部行为、公共路径和公共数据契约；
3. `HARNESS_ARCHITECTURE.md` 只能在不改变 PRD 契约的前提下定义实现架构和稳定性协议；
4. `PROJECT_PLAN.md` 只管理进度、阶段范围、依赖和交接，不改变产品行为。

出现冲突时，较低层文档必须先向较高层契约对齐；若高层文档本身存在歧义，再暂停相关开发并由项目负责人确认。开发人员或 Agent 不得静默选择解释。

### 2.1 开发准备度结论

当前内容足够 Codex 开始搭建 Harness 核心和 Codex Skill 适配骨架，但不足以直接完成具备真实领域判断能力的 MVP。

开发可以并行推进两条主线：

1. **工程主线**：搭建框架无关 Harness、项目数据契约、解析器、状态管理和导出框架；
2. **业务主线**：准备准则原文、审核后的条款拆解与映射、脱敏测试材料及人工期望输出。

业务材料未准备完成前，工程主线使用模拟规则和测试夹具，不得自行编造正式准则内容。

## 5. 工作流与责任边界

### WS-0：业务测试基准

**负责人建议**：准则专家、报告团队。

**工作内容**：

- 收集和确认准则版本；
- 制作人工拆解和映射样例；
- 准备脱敏客户材料；
- 制作期望母版、回应矩阵和缺口清单。

**禁止事项**：未经专业人员确认的 AI 拆解不得标记为正式规则。

### WS-1：Harness 基础

**负责人建议**：开发人员或 Codex。

**工作内容**：

- 工程初始化；
- `project.yaml` 模式和校验；
- 项目状态机；
- 公共错误和日志格式；
- JSON/JSONL 数据模型；
- 单元测试和测试夹具框架。

### WS-2：资料解析与证据库

**工作内容**：

- Word、PDF、Excel 解析；
- 文件哈希和增量复用；
- 证据位置、年份、单位及表格元数据；
- 解析失败和扫描 PDF 降级处理。

### WS-3：准则库与统一映射

**工作内容**：

- 准则版本注册和锁定；
- 原始条款、可检查要求和统一披露要求；
- 映射类型、审核状态和变更记录；
- 项目级自定义要求；
- 准则并集完整性检查。

### WS-4：母版、评价和导出

**工作内容**：

- 动态目录；
- 内容块和四类内容状态；
- 母版粗稿；
- 回应矩阵、缺口清单和证据清单；
- 内部版与干净版导出门禁；
- Word 和 Excel 导出。

### WS-5：适配、Agent 和试用

**工作内容**：

- 准则适配稿和差异清单；
- Codex Skill；
- Claude Code、WorkBuddy、Trae 的通用 Skill 接入说明；
- 公共状态和独立进程连续性测试；
- 试用指标记录和复盘。

## 7. 首轮开发范围

建议另一个 Codex 对话的首轮任务仅覆盖 `M1 工程骨架可运行`。M1 应优先形成一个 Garden 风格、可复制安装的 Skill Harness，而不是通用后台服务。

### 7.1 首轮必须交付

- 初始化可测试的 Harness 工程；
- 建立 `skills/sustainability-report-harness/` 自包含 Skill 包；
- 创建精简 `SKILL.md`、`manifest.json`、分阶段 `references/` 和确定性 `scripts/`；
- 定义标准项目目录；
- 实现 `project.yaml` 结构和校验；
- 实现 `disclosure_ledger.jsonl` 模式、唯一 ID 和引用完整性校验；
- 实现 PRD 中的项目状态机；
- 定义 `StandardVersion`、`Requirement`、`UnifiedDisclosure`、`Evidence`、`DisclosureContent` 和 `Assessment` 模型；
- 提供统一命令或程序接口完成项目创建、状态读取和配置校验；
- 建立单元测试、示例项目和开发说明；
- 创建模拟准则和示例项目，但不得写入虚构的正式准则知识；
- 建立“生成、校验、修复、再校验”的 QA 协议骨架。

### 7.2 首轮不做

- 不抓取或自行编造准则内容；
- 不实现未经测试材料验证的完整报告生成；
- 不同时适配多个 Agent；
- 不提前建设 SaaS、数据库服务或网页界面；
- 不把模拟规则标记为正式规则。

### 7.3 首轮完成标准

- 新项目可以被创建并产生符合 PRD 的目录；
- 无效配置能返回清晰错误；
- 项目状态可以持久化并恢复；
- Evidence、Outline、Anchor 和 Export Checkpoint 状态可以持久化，且不能绕过；
- 数据模型可以序列化和校验；
- 示例项目和测试全部通过；
- 后续解析器、规则库和生成模块可以通过稳定接口接入。

## 8. 开发过程管理

### 8.1 状态更新规则

每完成一个里程碑，开发对话应更新本文档：

- 将里程碑状态改为“进行中”“已完成”或“阻塞”；
- 记录实际交付文件；
- 记录测试结果；
- 记录新增决策和已知限制；
- 若修改 PRD 行为，必须同步更新 [PRD.md](./PRD.md)。

### 8.2 决策记录格式

新增决策统一追加到“已确认的关键决策”，使用下一个 `D-xx` 编号，并记录日期和原因。不得只把重要决定留在聊天记录中。

### 8.3 阻塞处理

- 缺少业务材料时，使用明确标记的模拟数据继续开发非领域模块；
- 不得用模型猜测代替准则专家确认；
- 阻塞真实验收但不阻塞工程开发时，应分别记录；
- 同一阻塞出现后，应指定负责人和需要的输入。

## 9. 风险与控制

| 风险 | 项目管理措施 |
| --- | --- |
| 范围过大导致迟迟无法试用 | 以 M1-M5 里程碑推进，每次只交付可验收闭环 |
| Skill 承载过多业务逻辑 | Skill 仅做交互和调用，状态及确定性逻辑放入 Harness |
| 没有真实规则就开始调模型 | 先做人工基准，模型输出只作为候选 |
| 多 Agent 同时开发造成接口漂移 | 先稳定公共契约和 Codex Skill，再增加第二个适配层 |
| 只关注生成速度忽视纠错成本 | 试用同时记录总工时、映射纠错和证据错误 |
| 业务决定只存在于对话 | 所有决定更新到 PRD 或本项目计划 |
