01 · Overview

它做什么，也不做什么

Harness 把分散的资料、准则要求、证据和草稿连接成一条可恢复、可审阅、可追溯的工作流。

它可以帮助你

  • 解析客户的 Word、文本型 PDF 和 Excel 材料
  • 形成多套准则要求的完整并集
  • 记录证据、缺口、推断和建议文本
  • 先审 Anchor 章节，再扩写完整母版
  • 从母版派生每套准则的适配稿
  • 在不同 Agent 或不同对话之间接续项目

它不会替你决定

  • 企业究竟适用哪套准则
  • 未经审核的法规文本是否正确
  • 推断能否升级为客户确认事实
  • 是否可以上传客户资料或调用云端 OCR
  • 报告是否满足法律、审计、鉴证或合规要求
当前产品状态 M5.3 工程功能已完成，但真实项目仍需要顾问提供正式、已审核的准则包，并完成专业审阅。仓库内的模拟准则只能用于测试。

02 · Get the Project

先从 GitHub 获取项目

仓库是公开的。你可以自己在终端克隆，也可以把明确的地址交给能够使用终端的 AI 助手完成。

A

手动命令行

适合习惯使用终端，或希望清楚看到每一步的用户。

cd ~/Documents
git clone https://github.com/AndrewJoey/sustainability-report-harness.git
cd sustainability-report-harness
make install

需要电脑已经安装 Git。如果不确定，可先运行 git --version。克隆只需执行一次，以后直接打开本地目录即可。

B

让 AI 助手克隆

适合在 Codex 或其他能够操作本地终端的 AI 助手中使用。

请把这个公开 GitHub 仓库克隆到我的本地工作区：
https://github.com/AndrewJoey/sustainability-report-harness.git

克隆完成后进入仓库，安装锁定依赖，并阅读 README.md 和完整的 sustainability-report-harness Skill。
先不要创建客户项目，不要提交或推送任何文件；完成后告诉我本地路径和检查结果。

AI 助手可能会请求联网和写入目标文件夹的权限。确认目标路径与仓库地址正确后再授权。

GitHub 仓库与客户项目是两件事 GitHub 只用于存放 Harness 代码。克隆完成后，真实客户项目和客户资料仍应创建在代码仓库之外，且不得提交或推送到 GitHub。

03 · Quick Start

选择你的使用方式

日常使用建议直接让支持本地 Skill 的 AI Agent 操作；手动命令更适合开发验证和问题排查。

把下面这段话发给 Agent

请确保 Agent 能读取本仓库及完整的 sustainability-report-harness Skill 目录。

请使用 sustainability-report-harness 创建一个可持续发展报告项目。
先向我确认客户材料、既有报告或模板、目标准则、参考案例、报告用途、读者、文风和必需议题。
不要直接开始生成；每个关键 Checkpoint 都等我确认。
客户资料默认只在本地处理，不要擅自联网、上传或调用云端 OCR。
接下来会发生什么？ Agent 会先问问题和创建项目目录，再引导你放入资料、确认准则、审阅证据与报告初稿。所有状态都写入项目文件，不依赖当前聊天记录。

先安装，再创建项目

需要 Python 3.11+ 和 uv。在仓库根目录安装锁定依赖：

make install

随后按本页的手动命令逐步执行。真实客户项目请放在本仓库之外。

05 · Workflow

一条有人工闸门的工作流

系统负责确定性校验，Agent 负责提出建议，顾问负责关键判断。任何 Checkpoint 都不应被跳过。

01

创建项目与确认数据授权

建立本地目录，明确联网、云端处理和保留边界。

需确认
02

确认项目问询

把资料、框架、参考案例和写作要求保存到 intake。

需确认
03

锁定准则并解析资料

只使用已审核准则包；DOCX、PDF、XLSX 转为可追溯证据。

需确认
04

形成要求并集与证据映射

保留每套准则的特有要求，标记冲突和信息缺口。

需确认
05

生成正式目录

根据要求并集、客户规格和证据状况提出目录。

需确认
06

先写 Anchor 代表章节

先校准结构、颗粒度、文风和证据表达，再批量写作。

需确认
07

完整母版与逐准则适配

母版覆盖要求并集，每份适配稿都引用母版内容 ID。

需确认
08

生成并验证 Markdown

输出母版、各准则适配稿和完整性清单。

导出确认
A

资料解析与 OCR

扫描 PDF 不会被悄悄跳过，也不会被自动上传。

如果 PDF 没有可搜索文本，Harness 会暂停，并检查本地是否有 PaddleOCR、Tesseract 等工具。若没有合适工具，由你选择：

  • 授权使用本地 OCR；
  • 明确授权其他方式或云端 OCR；
  • 补充可搜索版本；
  • 人工录入；
  • 暂停；
  • 仅对明确非关键来源记录为信息缺口。
你要确认：允许采用哪一种处理方式，以及文件是否可以离开本地环境。
B

证据、目录与 Anchor

不要等完整报告写完后才发现方向不对。

要求与证据映射先由 Agent 提议，全部新判断从 unreviewed 开始。顾问确认后，系统才会生成正式目录和一个代表性 Anchor 章节。

你要确认：证据是否支持结论、缺口是否真实、目录是否完整，以及 Anchor 的结构、文风和颗粒度是否适合批量扩写。
C

母版、适配与导出

一份业务真相源，避免多份报告互相失联。

完整正文从 state/disclosure_ledger.jsonl 派生。适配稿只能对母版内容执行保留、压缩、重组、补充或省略，并保留来源关系。

你要确认：完整母版和每套准则的适配动作。导出后如任何输入或输出改变，必须重新验证或生成。

06 · Deliverables

如何读懂生成结果

Markdown 初版会保留真实性标记和内部追溯 ID。它是一份“可审阅初版”，不是可以跳过专业复核的最终报告。

MD
outputs/markdown/master_report.md覆盖所选准则要求并集的母版
MD
outputs/markdown/adapted_<standard-id>.md每个已确认准则各生成一份适配初稿
#
outputs/markdown/report_manifest.json绑定配置、问询、准则锁、证据、账本、目录和所有输出的哈希

正文中的三类显式标记

[待确认-推断]

基于现有材料作出的推断，尚未被客户或顾问确认。

[建议文本]

建议采用的表达，不代表已经发生或已经获得证据支持。

[信息缺口]

当前资料不足，应补充材料、问询客户或明确披露限制。

HTML 注释不要随意删除 Markdown 内部的内容 ID、证据 ID 和适配来源 ID 用于追溯与后续校验。若要清理为客户版，应通过正式导出流程处理。

07 · Guardrails

五条必须遵守的边界

这些不是提示建议，而是为了保护客户资料、报告可信度和顾问判断设置的产品边界。

1

客户事实必须有客户证据

同行材料不能证明客户做过什么。

2

准则必须由专业人员审核

Agent 可以推荐报告期对应版本，但不能自行确认适用性或编造法规文本。

3

默认不联网、不上传

只有在项目数据授权明确允许时，才能使用外部处理或联网能力。

4

不跳过人工 Checkpoint

所有新语义判断先进入待审核状态，批准只记录真实发生的人工决定。

5

不直接覆盖人工修改

项目文件是事实来源；接续时先校验当前状态和完整性快照。

08 · Manual Reference

手动操作速查

下面用于开发验证和故障排查。把示例中的绝对路径、项目 ID、名称、日期和顾问姓名替换为真实值。

1. 创建客户项目 · 真实项目请放在代码仓库之外
uv run python skills/sustainability-report-harness/scripts/scaffold_project.py \
  /absolute/path/to/client-project \
  --project-id example-project \
  --project-name "示例项目" \
  --client-name "示例客户" \
  --period-start 2025-01-01 \
  --period-end 2025-12-31
2. 确认数据授权并进入项目规格问询
uv run python skills/sustainability-report-harness/scripts/workflow.py \
  /absolute/path/to/client-project transition awaiting_data_consent

uv run python skills/sustainability-report-harness/scripts/workflow.py \
  /absolute/path/to/client-project checkpoint data_consent approved \
  --approved-by "顾问姓名"

uv run python skills/sustainability-report-harness/scripts/workflow.py \
  /absolute/path/to/client-project transition awaiting_spec_confirmation
3. 保存并确认 intake · 先填写模板副本
uv run python skills/sustainability-report-harness/scripts/confirm_intake.py confirm \
  /absolute/path/to/client-project \
  /absolute/path/to/intake.json \
  --confirmed-by "顾问姓名"
4. 锁定一套已审核准则包 · 每套确认准则分别执行
uv run python skills/sustainability-report-harness/scripts/standards.py lock \
  /absolute/path/to/client-project \
  --package /absolute/path/to/reviewed-standard.json \
  --confirmed-by "顾问姓名"
5. 生成并验证 Markdown · 仅在前置 Checkpoint 全部满足后执行
uv run python skills/sustainability-report-harness/scripts/export_markdown.py generate \
  /absolute/path/to/client-project

uv run python skills/sustainability-report-harness/scripts/export_markdown.py validate \
  /absolute/path/to/client-project
6. 开发检查 · 在仓库根目录运行
make test
make lint
make validate
中间阶段没有“一条命令自动跑到底” 要求并集、证据、目录、Anchor、完整母版和适配都包含语义判断与人工审阅。日常使用应让 Agent 读取 Skill 后按项目当前状态推进。

09 · FAQ

常见问题

如果项目中断或结果看起来不对，先从状态、确认点和输入变化开始检查。

我只想快速生成一版报告，可以跳过前面的确认吗？
不建议，也不允许绕过强制 Checkpoint。最短可靠路径仍然需要确认数据授权、项目规格、准则、证据、目录、Anchor、母版和导出。这样做是为了避免把推断写成事实或遗漏准则要求。
优秀报告案例是必需的吗？
不是。你可以明确选择“不提供”。如果提供，要说明它用于风格参考、质量标杆或两者；其内容不能作为客户事实证据，也不能复制有辨识度的原文表达。
扫描 PDF 没有文字怎么办？
Harness 会暂停并列出可用选择。它不会静默安装 OCR、调用云服务或假装解析成功。由你决定本地 OCR、经授权的其他方式、补充可搜索文件、人工录入、暂停或对明确非关键来源记录缺口。
换一个 Agent 或重新打开对话，需要从头开始吗？
不需要。项目状态保存在本地目录中。新的 Agent 应读取 project.yaml、state/workflow.json 和 state/handoff.json，校验快照后从当前阶段继续。
为什么修改材料后输出提示过期？
输出清单会绑定客户材料、模板、参考案例、intake、准则锁、要求并集、账本、目录和生成文件的哈希。任何关联文件变化都可能使旧输出失效；请重新完成受影响的审阅并生成、验证输出。
可以直接生成 Word 吗？
当前 MVP 的主交付是 Markdown。仓库保留旧的 DOCX/XLSX 内部审阅能力，但 Word 模板化输出属于后续增强，不应作为当前默认交付来承诺。
